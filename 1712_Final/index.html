1. Database Normalization and Data Modeling

Normalization Concepts:

  • 1NF (First Normal Form):
    • Ensure every cell in a table contains only a single, indivisible value. (e.g., split full name into first and last name)
    • Identify and remove repeating groups (e.g., lists of phone numbers in one field).
  • 2NF (Second Normal Form):
    • Make sure the table is already in 1NF.
    • Remove partial dependencies where non-key attributes depend on only a part of a composite primary key.
    • Practice breaking down tables by isolating groups of attributes that belong with the keyâ€™s individual parts. (e.g., separate order details from order items)
  • 3NF (Third Normal Form):
    • Once in 2NF, eliminate transitive dependencies, meaning no non-key attribute should depend on another non-key attribute.
    • Exercise: Given a sample table, identify determinants (e.g., PROJ_NUM â†’ PROJ_NAME) and split tables to remove these indirect dependencies.

Data Modeling Practice:

  • Create Entity-Relationship Diagrams (ERDs) to visualize entities and relationships.
  • Define primary keys, candidate keys, and foreign keys.
  • Work on problems that require you to transform a logical data model into a physical one.

2. System Acquisition Strategies

Options Overview:

  • Custom In-House Development:
    • Benefits include complete alignment with business needs and retaining technical skills.
    • Risks include higher upfront cost, longer development time, and ongoing maintenance challenges.
  • Packaged Software:
    • Often used for common functions like payroll or accounting.
    • Customization is possible but may introduce limitations or require workarounds.
    • Consider compatibility with your current systems and whether the vendorâ€™s solution forces you to change business processes.
  • Outsourcing:
    • Involves hiring external vendors or service providers (including ASP and SaaS models).
    • Weigh benefits like lower initial development investment against risks such as loss of control and dependency on external expertise.

Gathering Information:

  • Understand formal tools like RFP (Request for Proposal), RFI (Request for Information), and RFQ (Request for Quote).
  • Practice outlining requirements, evaluation criteria, and timelines, and then comparing pros and cons based on vendor responses.

3. Architecture Design

Core Components and Models:

  • Components:
    • Software (data storage, application logic, presentation logic)
    • Hardware (servers, client machines, networking devices)
    • Networks (LAN, WAN, internet connections)
  • Architectural Models:
    • Client-Server Architecture:
      • Understand the distinction between thick clients (rich application logic on the client) and thin clients (minimal logic, mainly presentation).
      • Identify how tasks and responsibilities are divided between client and server.
    • Multi-Tier Architectures:
      • Two-Tier: Usually one tier for the client and one for the server (data storage and business logic).
      • Three-Tier: Adds an application server tier to handle business logic separately.
      • N-Tier: Further separation of concerns with more layers, increasing scalability and maintainability.
    • Other Approaches:
      • Server-based architectures (centralized processing vs. client-side processing) and zero/ultrathin clients.
      • Explore the benefits of virtualization and how cloud computing (public, private, hybrid) supports modern architecture with elastic scalability.

Practice Tips:

  • Draw diagrams showing the flow of data between tiers and map out where application logic, data access, and presentation layers are implemented.
  • Consider design trade-offs: network speed, load balancing, fault tolerance, and integration challenges.

4. Interface Design and Usability

Usability Fundamentals:

  • Key Metrics:
    • Effectiveness: Does the interface allow users to accomplish their tasks?
    • Efficiency: Are tasks completed with minimal effort?
    • Satisfaction: Do users enjoy interacting with the system?

The Design Process:

  • Conduct user research by developing personas and analyzing use scenarios.
  • Create low-fidelity prototypes such as sketches or wireframes, and evolve these into high-fidelity interactive prototypes.
  • Practice heuristic evaluations using established guidelines.

Golden Rules of Interface Design:

  • Focus on consistency (visual and functional), provide clear and immediate feedback, design for error prevention and easy error recovery, support reversible actions, and reduce cognitive load.

Practical Exercises:

  • Sketch different UI layouts and simulate user interactions.
  • Review case studies or examples (like booking a ticket online) to identify usability strengths and areas for improvement.

5. Program Design and Modularization

Modular Design Concepts:

  • Top-Down Approach:
    • Start by outlining a high-level diagram showing the main system components.
    • Break the system into smaller modules, identifying control modules (or drivers) and subordinate modules.
  • Cohesion and Coupling:
    • Cohesion: Ensure each module performs a single, well-defined task.
    • Coupling: Modules should interact as little as possible with each other to minimize dependencies.
    • Practice identifying and refactoring code examples that demonstrate high coupling or low cohesion.

Translating DFDs to Structure Charts:

  • Begin with Data Flow Diagrams (DFDs) to capture the business processes and then map these to a hierarchy of modules in a structure chart.
  • Learn to represent control flow, iterations (loops), and conditional processing in your charts.

6. Data Storage Design and Performance Optimization

Storage Formats and Models:

  • Files:
    • Understand when a flat file (like CSV) is sufficient vs. when a more robust database is needed.
    • Know the differences between types of files such as master, transaction, audit, and history files.
  • Databases:
    • Relational Databases: Focus on tables, primary/foreign keys, normalization, and SQL operations (CRUD).
    • NoSQL Databases: Get familiar with scenarios where document-oriented, wide-column, or graph databases are advantageous.
    • Multidimensional Databases: Often used in data warehousing and business intelligence with star schemas.

Logical to Physical Model Transition:

  • Identify how entities become tables.
  • Practice designing relationships (1:M, 1:1, and M:N) and implementing them through foreign keys and associative (bridge) tables.

Optimization Strategies:

  • For Storage Efficiency:
    • Emphasize normalization to reduce redundancy and eliminate null values, safeguarding data integrity.
  • For Access Speed:
    • Learn techniques such as denormalization (reintroducing controlled redundancy for faster read operations), clustering related data, and indexing to accelerate queries.

Practice Ideas:

  • Estimate table sizes and growth rates based on given data specifications.
  • Develop sample queries and analyze the impact of normalization vs. denormalization on performance.
